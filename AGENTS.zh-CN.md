# AGENTS.md

## 目的

本文件定义了 Godot 项目在“人类 + AI”跨多轮、跨多工具协作开发时的工作约定。

## 核心原则

使用那个最简单、最安全、最可复现、并且反馈清晰的工具来完成任务。

尽量优先采用文本优先、易于 diff 审查的工作流。只有当任务确实依赖编辑器状态、inspector 配置或视觉验证时，才使用 Godot 编辑器。

## 工具选择策略

### 优先顺序

1. 对 GDScript、JSON、配置文件、文档，以及可安全文本编辑的场景/资源文件，优先直接文本编辑。
2. 只有当当前环境中的具体集成已被证明确实可靠时，才使用 Godot MCP 风格工具进行结构化检查、编辑器启动或自动化。
3. 交互式编辑器工作使用标准 Godot 编辑器可执行文件。
4. 命令行诊断、验证、脚本和终端可见输出使用 Godot 控制台可执行文件。

### 直接文本编辑

以下内容优先直接文本编辑：

- 脚本
- 游戏玩法数据
- 文档
- 项目设置
- 可安全编辑的 `.tscn` / `.tres`

如果任务完全是文本型且可以通过 diff 清晰审查，就不要打开 Godot。

### 标准编辑器使用场景

以下工作使用标准 Godot 编辑器：

- inspector 修改
- 拖拽式场景连线
- 导入流程
- 动画设置
- Tile 或地图编辑
- 锚点与布局调整
- 视觉验证

### 控制台使用场景

以下工作使用 Godot 控制台可执行文件：

- 版本检查
- CLI 诊断
- 脚本执行
- 自动化验证
- CI 友好型工作流
- 终端可见日志与报错

### MCP 使用场景

以下工作使用 Godot MCP 风格工具：

- 启动编辑器
- 项目检查
- 调试输出捕获
- 在已验证稳定的前提下进行结构化自动化

不要默认认为所有 MCP 场景或节点编辑工具都安全可用。依赖它们之前先验证工具行为。

## 编码规则

- 在可行时尽量使用带类型的 GDScript。
- 保持函数聚焦、脚本职责单一。
- 默认优先组合，但当它明显更符合 Godot 场景或类模型时可以使用继承。
- 优先使用 `@onready`、导出引用或显式依赖设置，而不是脆弱的深层节点查找。
- 使用信号来实现解耦和清晰的事件式通信。
- 对始终需要的资源使用 `preload()`，对动态资源使用 `load()`。
- 对缺失资源和无效内容数据提供明确错误或安全回退。
- 保持命名一致：
  - 文件、变量、函数：`snake_case`
  - 类：`PascalCase`
  - 常量：`ALL_CAPS_SNAKE_CASE`
  - 信号：`snake_case`

## 协作规则

### 更适合 AI 先做的工作

- GDScript
- JSON 和内容数据
- 文档
- 架构说明
- 以文本形式完成的安全场景脚手架
- 可确定性的重构

### 更适合人类先做的工作

- 重度 inspector 配置
- 拖拽式场景连线
- 视觉布局微调
- 动画时间线编辑
- Tile 绘制
- 导入问题排查
- 依赖手感的验证

### 适合共同参与的工作

- 架构决策
- 场景边界划分
- 玩法方向
- 测试与验证策略
- 发布准备状态

## Session 协议

### 每次会话开始时

1. 阅读项目说明。
2. 查看当前重点和最近变更。
3. 为当前任务选择最简单且合适的工具。
4. 判断任务是文本优先还是编辑器依赖。

### 每次会话结束前

1. 记录已完成内容。
2. 记录下一步重点。
3. 记录仍需人工在编辑器中完成的操作。
4. 记录阻塞项或环境限制。
5. 在适合时提交完整且连贯的工作成果。

## 人工交接规则

当任务开始变得更依赖编辑器、视觉判断或 inspector 时，在一个干净检查点停下来，并记录：

- 已准备好的具体文件
- 需要执行的具体编辑器操作
- 预期成功标准
- 下一次会话需要验证的内容
