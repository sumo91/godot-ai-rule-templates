# CLAUDE.md

## Godot 开发策略

在处理 Godot 项目时，始终选择那个最简单、最安全、最可复现、并且反馈最清晰的工具来完成任务。

### 工具选择

- 对于 GDScript、JSON、配置文件、文档，以及可安全文本编辑的 `.tscn` / `.tres` 变更，优先直接编辑文本。
- 如果任务完全是文本型工作，而且可以通过 diff 清晰审查，就不要启动 Godot。
- 只有当 Godot MCP 风格工具确实带来明确优势时才使用它们，例如结构化检查、启动编辑器、调试输出或可靠自动化。
- 不要假设所有 MCP 场景或节点编辑工具都天然安全或适合当前任务。使用前先确认具体工具行为。
- 对于交互式编辑器工作，优先使用标准 Godot 编辑器可执行文件，尤其是 inspector 修改、拖拽式场景连线、导入流程、动画设置、Tile 编辑、布局调整和视觉验证。
- 对于命令行工作流，例如版本检查、诊断、脚本执行、验证、CI 任务和终端可见日志，优先使用 Godot 控制台可执行文件。

### 编码规则

- 在可行时尽量使用带类型的 GDScript。
- 保持函数聚焦、脚本职责单一。
- 默认优先组合，只有在确实符合 Godot 的场景或类模型时才使用继承。
- 优先使用 `@onready`、导出引用或显式依赖注入，而不是脆弱的深层节点查找。
- 使用信号来实现解耦和事件式通信。信号命名要清晰，并且只传递监听者真正需要的数据。
- 对始终需要的资源使用 `preload()`，对动态或可选资源使用 `load()`。
- 对缺失资源和无效内容数据提供明确错误或安全回退。
- 保持命名一致：
  - 文件、变量、函数使用 `snake_case`
  - 类使用 `PascalCase`
  - 常量使用 `ALL_CAPS_SNAKE_CASE`
  - 信号使用 `snake_case`

### 场景与架构规则

- 场景树尽量保持浅层。
- 可复用逻辑放在脚本里，可复用组合放在场景里。
- 只有当继承结构真的稳定时才使用场景继承。
- 如果文本脚本或数据文件更清晰，就不要把核心玩法逻辑藏进只靠 inspector 配置的内容里。
- 当项目确实能从迭代速度和可审查 diff 中受益时，优先考虑将职业、物品、敌人、区域和成长系统做成数据驱动定义。

### 人工交接规则

当任务开始变得更依赖编辑器、更依赖视觉判断、或更依赖 inspector 时：

- 在一个干净的检查点停下来
- 记录已经准备好的具体文件
- 记录接下来需要执行的具体编辑器操作
- 记录预期成功标准
- 记录编辑器步骤完成后下一次会话需要验证的内容

### Agent 行为规则

- 不要为了本可直接文本编辑的变更去使用编辑器。
- 不要盲目使用 MCP 自动化；先确认具体工具和操作是否适合当前任务。
- 对于脚本化或诊断型工作流，优先使用控制台可执行文件。
- 对于交互式或视觉型工作流，优先使用标准编辑器可执行文件。
- 当任务从“适合文本安全处理”跨入“编辑器拥有主导权”的阶段时，应停下来并进行清晰交接，而不是强行自动化。
